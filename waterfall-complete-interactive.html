📚 Introduction: Understanding SDLC Models

The Software Development Life Cycle (SDLC) is like a recipe for creating software. Just as there are many ways to bake a cake (from scratch, using a mix, or modifying a recipe), there are many different approaches to developing software. Each SDLC model provides a structured framework for planning, creating, testing, and deploying software systems.

🎯 Real-World Analogy

Think of SDLC models like different strategies for completing a major school project. Some students plan everything in detail before starting (Waterfall), others create a rough draft and keep improving it (Evolutionary), and some work in small chunks with frequent check-ins (Agile). Each approach has its strengths depending on the project type and requirements.

There are many models and variants of the SDLC frameworks or methodologies. Understanding when to use each model is crucial for successful software development. Let's explore the major SDLC models in detail.

🌟 Generic Life Cycle Models: The Big Picture

Before diving deep into the Waterfall model, let's understand the landscape of SDLC methodologies. Each model was designed to address specific challenges in software development.

🌊 Waterfall Model

Focus: Complete and correct planning to guide large projects and risks into successful and predictable results.

Best For: Projects with well-defined requirements that are unlikely to change, such as developing systems for regulated industries or replacing existing manual systems.

Key Characteristic: Sequential phases that flow downward like a waterfall—each phase must be completed before the next begins.

⛲ Fountain Model

Focus: A highly iterative approach that is well-suited to object-oriented software development. The fountain model allows for the fact that there is considerable overlap of activities throughout the development cycle.

Best For: Object-oriented projects where requirements evolve and phases naturally overlap.

Key Characteristic: Like water flowing in a fountain, development activities cycle back and forth, allowing for continuous refinement.

🌀 Spiral Model

Focus: Based on the unique risk patterns of a given project, the spiral model guides a team to adopt elements of one or more process models such as incremental, waterfall, or evolutionary prototyping.

Best For: Large, complex, high-risk projects where risk management is critical.

Key Characteristic: Combines iterative development with systematic risk analysis in each cycle (quadrant).

⚡ Rapid Prototyping

Focus: When a working model of software with some limited functionality is created for user interface. Prototyping is used to allow the users to evaluate developer proposals and try them out before implementation.

Best For: Projects where user interface and user experience are critical, or when requirements are unclear.

Key Characteristic: Quick creation of working models to gather user feedback before full development.

🏃 Agile Model

Focus: An iterative approach is taken and working software is built and delivered after each iteration. Each build is incremental in terms of features; the final build holds all the features required by the customer.

Best For: Projects with changing requirements and need for frequent customer feedback.

Key Characteristic: Short development cycles (sprints) with working software delivered frequently.

📈 Incremental Model

Focus: This model focuses on limited project scope and expanding or improving products by multiple iterations.

Best For: Projects that can be divided into smaller modules or when early delivery of core functionality is needed.

Key Characteristic: Software is developed and delivered in small, functional increments.

💡 Key Insight

There are many models of the software development life cycle including waterfall approach, evolutionary development including prototyping, fountain approach, formal transformation, and reuse-oriented approach according to your syllabus. The choice of model depends on project requirements, risk tolerance, team experience, and customer involvement.

🌊 The Waterfall Model: Deep Dive

The oldest and most linear life cycle model is the Waterfall model, which states that the phases are organized in a sequential unbreakable order. A project begins with feasibility analysis. Upon successful demonstration of the feasibility analysis, the requirements analysis phase begins.

🏗️ Building a House Analogy

Imagine building a house. You can't install the roof before building the walls, and you can't build the walls before laying the foundation. Similarly, in the Waterfall model, you must complete requirements analysis before design, design before coding, and coding before testing. Each phase builds upon the previous one, and going back is costly and difficult—just like trying to change the foundation after the walls are up!

The Five Phases of Waterfall

The figure below demonstrates the steps involved in the waterfall life cycle model. The design starts after the requirements analysis is done and coding begins after the design is done. Once the programming is completed, the code is integrated and testing is done. After this the regular operation and maintenance of the system takes place.

1. Requirement Analysis & Definition

What Happens: The developer analyzes user requirements and performs an investigation into the requirements, which produces developer version of requirements. Since users are often unable to describe what they need with precision, the developer finally gets the users to accept the proposal they can now begin.

Deliverable: Requirements document that both users and developers agree upon.

Real-World Example: Like writing a detailed shopping list before going to the grocery store, specifying exactly what you need, how much, and why.

2. System & Software Design

What Happens: This phase focuses on high level design: what programs are needed and how they will interact with each other. It also includes low level design (how the modules will work), interface design (what the interface will look like), and data design (what data is required). Hence the overall system architecture is defined.

Deliverable: System architecture document, interface designs, and data structure specifications.

Real-World Example: Like creating a detailed blueprint for your house, showing where each room goes, how they connect, and what materials you'll need.

3. Implementation & Unit Testing

What Happens: In this phase, designs are translated into code. Programs are written using a programming language or code generators.

Deliverable: Working code modules that have been individually tested.

Real-World Example: Like actually building the house according to the blueprint, one room at a time, and checking each room individually.

4. Integration & System Testing

What Happens: In this phase, the system is tested: each module is tested separately and in detail, then the system's modules work, the system works as intended with the expected data and that it does what the user requires.

Deliverable: Fully tested and verified system ready for deployment.

Real-World Example: Like checking if all the rooms work together properly—does the plumbing connect correctly? Does electricity flow to all rooms? Does everything meet building codes?

5. Operation & Maintenance

What Happens: In this phase, the system is maintained since the software will undergo change once delivered to the customers. These changes may stem from user input values into the system.

Deliverable: Updated system with bug fixes and enhancements based on user feedback.

Real-World Example: Like maintaining your house after moving in—fixing leaks, repainting walls, upgrading appliances based on your needs.

🔑 Key Concept: Deliverables and Sequential Flow

The waterfall model is the first to clearly identify the end of a phase and beginning of the others. Some documentation mechanism has to be employed at the end of each phase. This is usually done by some verification and validation. Validation means confirming the output of a phase is complete, with its input (which is the output of the previous phase) and that the output of the phase is consistent with overall requirements.

Important: In the waterfall approach, the result of each phase is called a deliverable or end product which flows sequentially into the next phase. For the coding phase, the output is the code.

✅ Advantages of the Waterfall Model

Strengths

  • Easy to explain to the user: The linear nature makes it simple to communicate the process to stakeholders.
  • Simple and easy to use: Straightforward methodology that's easy to understand and implement.
  • Stages and activities are well-defined: Clear milestones and deliverables at each phase.
  • Verification at each stage ensures early detection of errors or misunderstanding: Each phase is processed and completed one at a time, allowing for thorough review.
  • Easy to manage: Due to the rigidity of the model, each phase has specific deliverables and a review process.
  • Works well for smaller projects: Where requirements are very well understood and unlikely to change.
  • Easy to understand, easy to use: The sequential nature aligns with how many people naturally think about processes.

📝 Student Project Analogy

Think of writing a research paper with clear steps: (1) Choose topic, (2) Research and outline, (3) Write draft, (4) Edit and revise, (5) Submit final version. This linear approach works great when you know exactly what you're writing about and the requirements won't change. However, if your teacher suddenly changes the topic requirements halfway through, you'd have to start over—that's the main limitation of Waterfall!

⚠️ Disadvantages and Limitations of Waterfall

Disadvantages

  • Adjusting scope during the life cycle can kill a project: Changes are difficult and expensive to implement once a phase is completed.
  • No working software is produced until late during the life cycle: Users don't see a working product until near the end.
  • High amounts of risk and uncertainty: Problems may not be discovered until the testing phase.
  • Very rigid: Difficult to accommodate changes in requirements.
  • Does not reflect the iterative nature of software development: Real software development often requires going back and forth between phases.

Limitations

  • Assumes requirements will not change: This is possible for systems designed to automate an existing manual system. The waterfall is not applicable for the development of new systems, where there may be hidden or unknown requirements.
  • All requirements must be completely specified before development begins: The waterfall model stipulates that all requirements must be completely specified before the development can begin.
  • Not suitable for complex projects: Where requirements are likely to evolve or are not fully understood at the start.

⚡ When NOT to Use Waterfall

The waterfall model assumes that the requirements will not change. This is possible for systems designed to automate an existing manual system. The waterfall is not applicable for the development of new systems, where there may be hidden or unknown requirements. If you're developing something innovative or experimental, consider using Spiral, Agile, or Evolutionary models instead.

⚡ Rapid Prototyping: Build Fast, Learn Fast

Rapid Prototyping is when a working model of software with some limited functionality is created for user interface. Prototyping is used to allow the users to evaluate developer proposals and try them out before implementation.

🎬 Movie Trailer Analogy

Think of rapid prototyping like creating a movie trailer before making the full movie. The trailer gives viewers a taste of what to expect, and based on their reactions, filmmakers can adjust the final product. Similarly, a prototype shows users what the software will look like and how it will work, allowing for feedback before investing in full development.

The Four Phases of Rapid Prototyping

Phase 1: Requirements Planning Phase

A workshop utilizing structured discussion of business problems.

Phase 2: User Description Phase

Automated tools capture information from users.

Phase 3: Construction Phase

Productivity tools, such as code generators, screen generators, etc. inside a toolbox are used to build the prototype.

Phase 4: Cutover Phase

Installation of the system, user acceptance testing, and user training.

Strengths and Weaknesses

Strengths

  • Reduced cycle time and improved productivity: With fewer people means lower costs.
  • Time box approach mitigates cost and schedule risk: Development is controlled within set time limits.
  • Customer involved throughout: Minimizes risk of not achieving customer satisfaction and business needs.
  • Uses modeling concepts: Information about business, data, and processes. The software object is completed in a shorter period of time and significantly cheaper. With rapid prototyping, you can have an idea about how the actual software will function as the requirements of the entire system.

Weaknesses

  • Accelerated development process must give quick responses to the user: Requires fast turnaround times.
  • Risk of never achieving closure: Continuous changes can prevent project completion.
  • Requires a system that can be modularized: Not all systems can be easily broken into modules.
  • Developers and customers must be committed to rapid-fire activities: Requires high engagement from all parties in an abbreviated time frame.
  • The word "rapid" is relative: Construction of a model with contemporary methods can take from several hours to several days.

🔄 Evolutionary Development: Grow Your Software

Software development is very interactive as user input and constant feedback play a critical role in the development before critical junctures in the development are reached. Evolutionary development is effective in the development of new systems, instead of creating comprehensive artifact, such as a requirements specification, that you review and accept in one go, you instead evolve the critical development artifacts over a period of time in an iterative manner.

📝 Essay Writing Analogy

Think about writing an important essay. You don't write it perfectly in one go. Instead, you write a rough draft, get feedback from friends or teachers, revise it, get more feedback, revise again, and keep improving until it's excellent. Evolutionary development works the same way—you create a basic version, get user feedback, improve it, and repeat until the software meets all needs.

The evolutionary development model divides the development cycle into smaller, incremental models. In each model, the users provide feedback on the product. The planning stage of the next model is dependent on the feedback from the previous model. Hence, the development team responds, often by changing the product, plans or process.

The evolutionary development approach aims to develop a mature system through a series of prototype iterations.

Strengths and Weaknesses

Strengths

  • Constant updates and changes: Due to the constant updates and changes that become available through the user input, the programs will mostly fit the desired need.
  • Flexibility: Allows for changes and improvements throughout the development process.
  • User involvement: Continuous user feedback ensures the final product meets actual needs.

Weaknesses

  • Extensive development process: The development process may be extensive due to the constant feedback.
  • Gap in functionalities: There can be a gap in the type of functionalities needed, between management, developer and end user.
  • Unclear endpoint: It may be difficult to determine when the system is "complete."

🌀 Spiral Model: Managing Risk Through Cycles

An SDLC model that can be likened to a combination of waterfall model and RAD prototyping with risk analysis. Each cycle involves the same sequence of steps as the waterfall process model. With risk analysis, the software is developed in phases called quadrants.

🎮 Video Game Development Analogy

Think about how video games are developed. Developers create an alpha version (high risk of bugs), then a beta version (medium risk, more features), then release candidates (low risk, almost complete), and finally the full release. At each stage, they assess risks, get player feedback, and decide whether to continue. If risks are too high, they might go back and redesign. This is exactly how the Spiral model works!

Strengths and Weaknesses

Strengths

  • The initial design does not have to be perfect: Users can be closely tied to all lifecycle steps, there is early and frequent feedback from users, and the overall costs assessed frequently.
  • Flexibility in design: Changes can be incorporated at any stage.
  • Risk management: Continuous risk assessment helps identify and mitigate problems early.
  • User involvement: Frequent user feedback ensures the product meets needs.

Weaknesses

  • Time spent for evaluating risks: It is too large for small or low-risk projects, also the time spent planning, resetting objectives, doing risk analysis and prototyping may be excessive.
  • Complex model: Where a risk assessment expertise is required and the spiral may continue indefinitely.
  • Requires expertise: Risk analysis requires highly specific expertise.
  • Can be expensive: Multiple iterations and extensive risk analysis increase costs.

🔬 Formal Transformation: Mathematical Precision

Formal transformation is a mathematically-based technique for the specification, development and verification of hardware and software systems.

It is similar to the waterfall model but its specification is converted to a mathematical module. It is a mathematical method used to specify a hardware and/or a software system.

It is often used to verify whether a specification is realizable or to mathematically prove properties of a system without necessarily running it.

🧮 Math Proof Analogy

Think about proving a mathematical theorem. Instead of just testing examples (like checking if 2+2=4, 3+3=6, etc.), you write a formal proof that works for ALL cases. Formal transformation does the same for software—it mathematically proves the software will work correctly in all scenarios, rather than just testing specific cases. This is incredibly reliable but requires advanced mathematical skills!

Phases of Formal Transformation

It is similar to the waterfall model as each phase has to be finished before moving on to the next. It is used especially when developing systems that require safety, reliability and security.

  1. Requirements Definition: The requirements for the system are formally expressed.
  2. Formal Specification: The software requirements and specifications are combined in this phase and are expressed mathematically.
  3. Formal Transformation: The design implementation and unit testing are done in this phase, using mathematical notation.
  4. Integration & System Testing: The system is deployed into the environment and is tested.
  5. Maintenance: The system is constantly monitored, upgraded and updated.

Strengths and Weaknesses

Strengths

  • The development model is described as precise and free of errors: Provides mathematical certainty of correctness.
  • Formal transformation is said to be error-free: Due to the tedious mathematical specifications which allows no room for errors. It is said that formal transformation ensures that the program or software runs as it is supposed to without any glitches that will affect the running of any vulnerable systems.
  • High reliability: Mathematical proofs ensure the system works as specified.
  • Safety-critical systems: Ideal for systems where failure could be catastrophic.

Weaknesses

  • Very costly: It is hard to communicate aspects to customers, it takes very long time to complete due to its need for accuracy. A very limited set of developers are intensively trained in this model and the other must be extensively trained in this model.
  • Communication difficulties: Hard to explain mathematical specifications to non-technical stakeholders.
  • Time-consuming: Requires extensive time for mathematical proofs and verification.
  • Limited expertise: Few developers have the necessary mathematical background.

⚠️ When to Use Formal Transformation

This model is best suited for safety-critical systems such as aircraft control systems, medical devices, nuclear power plant controls, or financial transaction systems where errors could have catastrophic consequences. For typical business applications, the cost and complexity usually outweigh the benefits.

⛲ Fountain Model: Iterative and Overlapping

As the iterative approach of the model that inspired the name, same as water in a fountain rises and falls back down to the pool below. It was developed to support object-oriented software development. The fountain model allows for the fact that there is considerable overlap of activities throughout the development cycle.

✏️ Essay Revision Analogy

When you write an important essay, you don't finish the introduction, then the body, then the conclusion in strict order. Instead, you might write a rough introduction, start the body, go back to improve the introduction, continue the body, start the conclusion, go back to fix the body, and so on. Everything overlaps and flows back and forth—just like water in a fountain! This is how object-oriented development works too.

The general movement from the analysis phase through the design phase and to implementation is interwoven with iterative cycles across theses phases during the software development process.

Strengths and Weaknesses

Strengths

  • Supports the iterative nature of software development: Within phases which allows for flexibility. It recognizes that there is an overlap of activities throughout the development cycle. It supports incremental development of system. It leads to focusing on sections of the whole system known as subsystems.
  • Flexibility: Allows developers to move between phases as needed.
  • Object-oriented friendly: Well-suited for OOP development practices.
  • Realistic approach: Reflects how development actually happens in practice.

Weaknesses

  • Limited risk management focus: There is not much focus on risk management. Some phases cannot begin without others. It slows down the development time of the software system.
  • Dependencies: Some phases still depend on others being completed first.
  • Can be slower: The iterative nature may extend development time.

💡 Fountain vs Waterfall

The key difference between Fountain and Waterfall is that Waterfall flows in one direction (top to bottom), while Fountain allows water to flow up and down repeatedly. In development terms, Waterfall requires you to complete each phase before moving to the next, while Fountain lets you move back and forth between phases as needed—much more realistic for complex software development!

♻️ Reuse-Oriented Development (ROD): Don't Reinvent the Wheel

The reuse-oriented model, also called reuse-oriented development (ROD). This model gives an approach to system development where preexisting code or components of commercial or open-source (OOS) systems are used.

📋 Template Analogy

Imagine you need to create a presentation for a school project. Instead of designing every slide from scratch, you use a PowerPoint template that already has nice layouts, color schemes, and formatting. You just fill in your content and customize what you need. That's exactly what Reuse-Oriented Development does—it uses existing, tested software components as building blocks rather than coding everything from zero!

The reuse-oriented model is not always practical in its pure form. The reuse-oriented model can reduce the time associated with software development when compared with more tedious manual methods.

It can also save time because each phase of the process builds on the previous phase which has already been refined. When carefully carried out, ROD can minimize the likelihood of errors or bugs making their way into the first beta version of a software product.

Phases of Reuse-Oriented Development

1. Requirements Definition

Initial requirements are gathered and defined.

↓

2. Component Analysis

Search for and analyze existing components that could be reused.

↓

3. Requirements Modification

Adjust requirements based on available components.

↓

4. System Design with Reuse

Design the system incorporating reusable components.

↓

5. Development & Integration

Develop new components and integrate with reused ones.

↓

6. System Validation

Test and validate the complete integrated system.

Strengths and Weaknesses

Strengths

  • Increased reliability: As components would have already been deployed and tested in a variety of problem domains. Errors would have been discovered earlier and eliminated. Hence reducing the risk of component than in the costs of development. This process has the potential to reduce the amount of development.
  • Reduced development time: Building on previous work saves time.
  • Standards compliance: Reusable components often follow industry standards.
  • Lower risk: Proven components have fewer bugs than new code.
  • Cost-effective: Less time and fewer resources needed for development.

Weaknesses

  • Increased maintenance costs: The inaccessibility of the component source code, will drive maintenance costs. Expensive. Finding and adapting reusable components can be tedious and very time consuming. Hence all engineers would want to be very confident about the compatibility of components.
  • Source code inaccessibility: Can't fix bugs in third-party components.
  • Compatibility issues: Components may not work well together.
  • Limited customization: Reused components may not perfectly fit needs.
  • Dependency risks: Relying on external components can be risky if they're discontinued.

🔍 Feasibility Study: Can We Actually Build This?

The feasibility study is focused towards the goals of the organization. This study analyses whether the project is practically created. The main aim of the feasibility study activity is to determine whether it would be financially and technically feasible to develop the product.

🚗 Car Purchase Analogy

Before buying a car, you ask yourself three questions: (1) Can I afford it? (Economic), (2) Can I actually drive it and maintain it? (Technical), and (3) Will it actually improve my life and meet my needs? (Operational). A feasibility study asks the same three questions about a software project before committing resources to build it!

The feasibility report is positive towards undertaking the project, next phase starts with gathering requirements from the users. The development team communicates with the stakeholders to know their ideas on what the software should provide and which features are to be included.

Types of Feasibility

1. Technical Feasibility

This outlines the technical aspects proposed to be used, which will give a technical solution and constraints on the system. This is concerned with specifying equipment and software that will successfully satisfy the user requirement.

Questions to Ask:

  • Do we have the technical expertise to build this?
  • Is the technology mature enough and available?
  • Can our current infrastructure support this system?
  • Are there technical risks that could prevent success?

2. Economic Feasibility

This considers the costs in relation to the benefits of the proposed system. The procedure is to determine the benefits and savings that are expected from a proposed system and compare them with costs. If benefits outweigh costs, a decision is taken to design and implement the system. Further justification or an alternative to the proposed system will have to be made.

Questions to Ask:

  • What are the development costs?
  • What are the expected benefits and savings?
  • Do benefits exceed costs?
  • What is the return on investment (ROI)?
  • Can we afford to build AND maintain this system?

3. Operational Feasibility

Operational feasibility determines if the human resources are available to operate the system once it has been installed.

Questions to Ask:

  • Do users actually want this new system?
  • Will users be able to use it effectively?
  • Does it fit into current business processes?
  • Will it solve the intended problem?
  • Are there organizational or cultural barriers?

📋 Requirement Gathering: Understanding What Users Really Need

If the feasibility report is positive towards undertaking the project, next phase starts with gathering requirements from the users. The development team communicates with the stakeholders to know their ideas on what the software should provide and which features are to be included.

Software Requirement Validation

After requirements are developed, the requirements will be validated. Users might ask for illegal features, or impractical requirement, or experts may find that the requirements can be checked against following conditions:

✅ Requirements Validation Checklist

Verification vs Validation

Aspect Verification Validation
Question Asked "Are we building the product right?" "Are we building the right product?"
Focus Process-oriented: Did we follow the specifications correctly? Product-oriented: Does it meet user needs?
Definition Checking that the software conforms to its specification Checking that the system meets its specific functional and non-functional requirements
Example Checking if code follows the design document Checking if the system solves the user's problem

🎯 Verification vs Validation Analogy

Verification: You're baking a cake following a recipe. Verification checks if you followed the recipe correctly—did you use the right ingredients in the right amounts?

Validation: Once the cake is baked, validation checks if it actually tastes good and if people want to eat it. You might have followed the recipe perfectly (verification ✓), but if the cake tastes terrible, it fails validation ✗!

Importance of Validation

🔑 Why Validation Matters

To correct errors in a requirements document can lead to significant costs when they are eventually discovered during development or deployment.

The cost of making a system change resulting from a requirement problem is much greater than repairing design or coding errors.

Finding and fixing a requirements error early can save 100x the cost compared to finding it after deployment!

Requirements Review

A requirements review is a manual process, which involves multiple readers from both the client and the contracted staff. It involves checking the requirements document for anomalies and omissions in the requirements that are to be implemented by the system.

Validation Techniques

1. Requirements Review

A team of reviewers analyzes the requirements systematically.

Benefit: Multiple perspectives catch more errors.

2. Prototyping

An executable model of the system is demonstrated to end-users and customers for them to experiment with and see if it meets their real needs.

Benefit: Visual demonstration helps users understand what they're getting.

3. Test-Case Generation

If a test is difficult or impossible to design, this usually means the requirements will be difficult to implement and should be reconsidered.

Benefit: Identifies implementation challenges early.

🎤 Requirements Elicitation Techniques: Gathering the Right Information

The following are tools and techniques used to carry out the feasibility study and requirements analysis in the analysis phase of the SDLC.

1. Requirement Elicitation Techniques

Requirement elicitation techniques such as interviews, questionnaires, observations and review of internal documents are used to gather information.

👥 Interviews

An interview is the questioning of a person or having a conversation in which information is elicited. Interviews are a strong medium to collect requirements. Organizations may conduct several types of interviews.

Reasons for Using Interviews

Interviews are a useful method to:

  • Investigate issues in an in-depth way
  • Discover how individuals think and feel about a topic and why they hold certain opinions
  • Inform decision making, strategic planning and resource allocation
  • Provide ventilation of sensitive topics which people may feel uncomfortable discussing in a focus group
  • Add a human dimension to impersonal data
  • Deepen understanding and explain statistical data collected in questionnaires

Advantages

  • They are useful to obtain detailed information about personal feelings, perceptions and opinions
  • They allow more detailed questions to be asked
  • They usually achieve a high response rate
  • Ambiguities can be clarified and incomplete answers followed up

Disadvantages

  • They can be very time-consuming: setting up, interviewing, transcribing, analyzing, feedback, reporting
  • They can be costly

📝 Questionnaires

Questionnaires are an inexpensive way to gather data from a potentially large number of respondents. Often they are the only feasible way to reach a number of reviewers large enough to allow statistical analysis of the results.

Reasons for Using Questionnaires

Questionnaires are a useful method to investigate:

  • Patterns, frequency, ease and success of use
  • User needs, expectations, perspectives, priorities and preferences
  • User satisfaction with collections and services
  • Shifts in user attitudes and opinions
  • Relevance of collections and services to user needs

Advantages

  • They are relatively easy to analyse
  • A large sample of the given population can be contacted at relatively low cost
  • They are simple to administer
  • The format is familiar to most respondents
  • They should be simple and quick for the respondent to complete
  • Information is collected in a standardized way
  • Respondents have time to think about their answers; they are not usually required to reply immediately

Disadvantages

  • If you forget to ask a question, you cannot usually go back to respondents, especially if they are anonymous
  • It is sometimes difficult to obtain a sufficient number of responses, especially from postal questionnaires
  • Respondents may ignore certain questions
  • Questionnaires may appear impersonal
  • Questions may be incorrectly completed

👀 Observations

The observation method of research is basically developed for observing people in their natural setting. It focuses more on their everyday normal life. There are two types of observation methods:

Non-Participant Observation

The researcher observes without participating in the activities.

Participant Observation

The researcher actively participates in the activities being observed.

Advantages

  • It helps in overcoming issues of validity
  • It is useful when the end-users cannot provide certain information
  • It is also useful when the subject is feared to have provided inaccurate information
  • Supplements other data collection methods
  • Simple

Disadvantages

  • Frequently measuring attitudes or opinions
  • Time and costs are high
  • Does not provide complete information for more complex jobs
  • May interfere with job performance
  • Validity and reliability may be problematic
  • May bias worker performance
  • Small sample size
  • Requires skilled observer

📄 Review of Internal Documentation

Document review is a way of collecting data by reviewing existing documents used inside the organization for the which the system is being developed.

The documents may be hard copy or electronic, and may include reports, invoices, regulation sheets, funding proposals, meeting minutes, newsletters, and purchase orders.

Reasons for Using Document Analysis

Document analysis is a useful method to investigate:

  • Decision making and strategic planning
  • Resource allocation
  • Outcomes alignment and library policy

Advantages

  • It overcomes the difficulties of encouraging participation by users
  • There are few costs involved other than staff time

Disadvantages

  • It is not suitable to evaluate user opinions, needs or satisfaction with services
  • Some documents may be sensitive and not accessible

🔧 Prototyping

Software prototyping, an activity during certain software development, is the creation of incomplete versions of the software program being developed.

A prototype typically simulates only a few aspects of the features of the eventual program, and may be completely different from the eventual implementation.

A prototype allows users of the proposed system to evaluate developers' proposals for the design and functionality of the eventual software artifact. Prototyping can also be used by end users to describe and prove requirements that developers have not considered.

Prototyping has several benefits: the software designer and implementer obtains early feedback from the specification provided to developers. Clients and contractors can compare the software to, to ensure that it is according to which the software program is built.

Benefits of Prototyping

  • Reduced time and costs: Prototyping can improve the quality of requirements and the end specification provided to developers.
  • Improved and increased user involvement: Prototyping requires user involvement and allows them to see and interact with a prototype allowing them to provide better and more complete feedback and specifications.
  • Specific feedback on system functionality: Prototyping allows the designer and implementer to obtain early feedback on the design, allowing them to see and interact with a prototype, allowing them to provide more complete and accurate requirements.

2. Additional Requirements Analysis Tools

The following are additional tools and techniques used to carry out requirements analysis:

3. Data Flow Diagram (DFD)

DFD is used to document the flow of information, to show the storage of data through a particular system.

Purpose: Visualizes how data moves through the system from input to storage to output.

4. Data Dictionary

Data Dictionary provides the name, type, size and description of all the relevant data used in the software system.

Purpose: Centralized repository of information about data elements.

5. Semantic Data Model

A semantic data model in software engineering is a data modeling technique for the context of its interrelationships with other data. A semantic data model is an abstraction which defines how the stored symbols relate to the real world. A semantic data model is sometimes called a conceptual data model.

Purpose: Represents the meaning of data and relationships in a way that's closer to how humans think.

6. Object Models

Object Models are used to store specific information/data about entities in a software system.

Purpose: Represents real-world entities as objects with properties and behaviors.

7. Computer Aided Software Engineering (CASE)

Computer Aided Software Engineering (CASE) tools are used to analyse the present data and model a software system.

Purpose: Automated tools that support various software engineering activities including requirements analysis.

Functional vs Non-Functional Requirements

Functional Requirements Non-Functional Requirements (Constraints)
Describes the interaction between the system and its environment Put restrictions on the system - describes a restriction on the system that limits our choices for constructing a solution to the problem
Describes how the system should behave, given certain inputs These constraints limit the selection of the programming language, platform, or implementation techniques or tools
Provides answers to questions that are independent of an implementation of a solution to the application's problem Put restrictions on the system
Example: "The system shall allow users to log in with username and password" Example: "The system must respond within 2 seconds" or "The system must use Python 3.11"

🎮 Gaming Console Analogy

Functional Requirements: What the console DOES - plays games, connects online, saves progress, supports multiplayer.

Non-Functional Requirements: HOW WELL it does it - loads games in under 10 seconds, supports 4K resolution, runs quietly, uses less than 200W power, costs under $500.

Issues in Requirement Elicitation

⚠️ Common Challenges

  • Ambiguous understanding of business processes: Stakeholders may not clearly understand or explain their own processes.
  • Conflicting stakeholder interests: Different users want different things.
  • Inconsistency within a single process: The same task might be done by multiple users in different ways.
  • Insufficient or incomplete input from stakeholders: Users don't provide enough detail.
  • Changes in system requirements after project has started: Requirements evolve during development.

📖 Glossary: Key Terms and Concepts

Master these essential terms to excel in software development life cycle methodologies:

SDLC (Software Development Life Cycle)

A structured process for planning, creating, testing, and deploying software systems.

Waterfall Model

A sequential SDLC model where each phase must be completed before the next begins, flowing downward like a waterfall.

Deliverable

The end product or output of a phase in the SDLC that flows into the next phase.

Verification

Checking that the software conforms to its specification. Asks: "Are we building the product right?"

Validation

Checking that the system meets user needs and requirements. Asks: "Are we building the right product?"

Prototype

An incomplete version of software created to demonstrate features and gather user feedback before full development.

Iteration

A cycle of development where software is repeatedly refined and improved based on feedback.

Incremental Development

Building software in small, functional pieces that are delivered and integrated one at a time.

Spiral Model

An SDLC model that combines iterative development with risk analysis, progressing through repeated cycles (quadrants).

Quadrant

A phase or cycle in the Spiral model where development and risk analysis occur.

Feasibility Study

Analysis to determine if a project is technically, economically, and operationally viable before committing resources.

Requirements Elicitation

The process of gathering requirements from stakeholders through interviews, questionnaires, observations, and other techniques.

Functional Requirements

Specifications describing what the system should do—its features, capabilities, and behaviors.

Non-Functional Requirements

Constraints on how the system operates, such as performance, security, usability, and platform requirements.

DFD (Data Flow Diagram)

A visual representation showing how data moves through a system from input to storage to output.

Data Dictionary

A centralized repository containing names, types, sizes, and descriptions of all data elements in a system.

Semantic Data Model

An abstraction that defines how stored data relates to the real world and its meaning in context.

CASE (Computer Aided Software Engineering)

Automated tools that support software engineering activities including analysis, design, and development.

ROD (Reuse-Oriented Development)

A development approach that uses preexisting code or components rather than building everything from scratch.

Formal Transformation

A mathematically-based technique for specifying, developing, and verifying systems with mathematical precision.

Fountain Model

An iterative SDLC model where development activities overlap and cycle back and forth, like water in a fountain.

Evolutionary Development

A development approach where the system evolves through multiple iterations based on continuous user feedback.

Rapid Prototyping

Quick creation of working models with limited functionality to gather user feedback before full implementation.

Agile Model

An iterative approach where working software is built and delivered after each short development cycle (sprint).

Technical Feasibility

Assessment of whether the required technology and expertise are available to build the system.

Economic Feasibility

Analysis comparing project costs against expected benefits to determine financial viability.

Operational Feasibility

Evaluation of whether users will accept and be able to effectively use the proposed system.

Requirements Review

A manual process where multiple reviewers check requirements documents for errors and omissions.

Stakeholder

Anyone who has an interest in or is affected by the software system—users, clients, developers, managers.

Object Model

A representation of real-world entities as objects with properties, behaviors, and relationships.
